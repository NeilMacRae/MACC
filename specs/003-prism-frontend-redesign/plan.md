# Implementation Plan: Prism Design System Migration & Navigation Redesign

**Branch**: `003-prism-frontend-redesign` | **Date**: 4 March 2026 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/003-prism-frontend-redesign/spec.md`

## Summary

Migrate the MACC frontend from custom Tailwind-based components to `@ecoonline/prism-web-components-react` and restructure navigation to consolidate Initiatives, Scenarios, Targets, and MACC Chart into a single tabbed MACC Modelling hub. Tab state encoded via `?tab=` query parameter (user-confirmed design decision). A prerequisite backend migration decouples `emission_targets` from `organisational_contexts` so targets are stored directly under `organisations` — this is delivered as Phase 0 before any frontend work begins. Target API paths remain unchanged.

### Phase 0: Backend — Decouple Targets from OrganisationalContext

**Goal**: Migrate `emission_targets` to be stored directly under `organisations`, removing the hard dependency on a context record existing before targets can be created.

**Scope**:
- New Alembic migration (`d2e3f4a5b6c7`): add `organisation_id` FK to `emission_targets`, backfill via join through `organisational_contexts`, drop `context_id` column
- Update `EmissionTarget` SQLAlchemy model: `context_id` → `organisation_id` FK to `organisations.id`
- Remove `targets` relationship from `OrganisationalContext`; add `targets` relationship to `Organisation`
- Update `target_service.py`: all CRUD functions query by `organisation_id` directly; remove the `_get_context` guard that blocked target access when no context existed
- Update `TargetResponse` Pydantic schema: `context_id` field renamed to `organisation_id`
- API paths remain unchanged (`/context/targets`)

**Test gate**:
- [ ] `alembic upgrade head` succeeds on a database with existing target data
- [ ] `alembic downgrade -1` succeeds (round-trip)
- [ ] Unit tests for `target_service`: `list_targets`, `create_target`, `update_target`, `delete_target`, `get_target_progress` all pass without requiring a context record
- [ ] Backend smoke test: GET `/context/targets` returns 200 (empty list) when no context exists
- [ ] Backend smoke test: POST `/context/targets` succeeds when no context exists
- [ ] All existing backend tests pass

**Files touched**: `alembic/versions/d2e3f4a5b6c7_decouple_targets_from_context.py`, `src/models/context.py`, `src/models/organisation.py`, `src/services/target_service.py`, `src/schemas/context.py`
**Risk**: Low-Medium — isolated backend change; API paths unchanged; no frontend impact until Phase 3
**User stories**: Prerequisite for FR-015 (Targets tab always available)

### Phase 2: MACC Modelling Hub — Tabbed Shell

**Goal**: Create the tabbed MACC Modelling page with `?tab=` query parameter routing.

- Create `MACCModellingPage.tsx` with three tabs: Initiatives, Scenarios, Targets

## Design Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Tab URL encoding | `?tab=` query parameter | User-confirmed. Simple, works with `useSearchParams`, supports bookmarking, avoids nested route complexity |
| Default tab | Initiatives (`?tab=initiatives`) | Per FR-011; the Initiatives tab is the unified MACC modelling workspace (chart + initiative management + AI suggestions) |
| Tab state preservation | Hidden mount (`display: none`) | Inactive tabs stay mounted preserving local state; TanStack Query handles data caching |
| Tab state on refresh | URL preserves tab selection only | Per FR-012/assumptions; form data is in-memory only |
| D3 chart approach | Keep custom, wrap in Prism container | Prism won't have a MACC chart; D3 integration is stable |
| Target storage | Direct FK to `organisations.id` | Decoupled from `organisational_contexts` so Targets tab is always functional regardless of context; aligns with spec Key Entities intent |
| Target API paths | Unchanged (`/context/targets`) | Avoid breaking existing frontend integrations; internal storage change only |
| Migration order | Phase 0 backend decoupling → Structure → Shell → Components → Pages | Backend prerequisite first; then structural changes (highest user impact), then systematic component replacement |
| Phase granularity | 8 phases + sub-phases in Phase 7 | Each independently testable; app fully functional after each |
