# Implementation Plan: Docker & PostgreSQL Formalisation

**Branch**: `004-docker-postgres-setup` | **Date**: 31 March 2026 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/004-docker-postgres-setup/spec.md`

## Summary

Containerise the MACC application with Docker and remove all SQLite code paths, formalising PostgreSQL as the sole database engine across all environments. This involves: creating multi-stage Dockerfiles for backend (Python/FastAPI) and frontend (React/Vite → nginx), authoring a `docker-compose.yml` with backend, frontend, `db`, and `test-db` PostgreSQL services, stripping SQLite-specific logic from `database.py`, `conftest.py`, and `alembic/env.py`, removing the `aiosqlite` dependency, and updating the Makefile to use `docker-compose exec` for `migrate` and `seed` targets. This implements Constitution Principle VIII (Dev/Prod Parity & Containerisation).

## Technical Context

**Language/Version**: Python 3.11+ (backend), TypeScript 5.6 / React 18.3 (frontend)
**Primary Dependencies**: FastAPI 0.134, SQLAlchemy 2.0.47 (async), Alembic 1.18, asyncpg 0.30, Vite 5.4, nginx (alpine)
**Storage**: PostgreSQL 15+ via asyncpg; managed by Railway in production, docker-compose locally
**Testing**: pytest + pytest-asyncio (backend), Vitest + RTL (frontend), Playwright (E2E)
**Target Platform**: Docker containers — docker-compose for local dev, Railway for production
**Project Type**: Web application (backend API + frontend SPA)
**Performance Goals**: API < 200ms p95; full stack startup via `docker-compose up` < 60s
**Constraints**: Single-command startup; dev hot-reload for both backend and frontend; deterministic builds via pinned deps
**Scale/Scope**: 2 Dockerfiles, 1 docker-compose.yml, 1 nginx.conf, ~8 existing files modified, ~3 files removed/replaced
